# WASM Crates in tugdeck

*Where Rust-compiled-to-WASM lives in this repo, the per-crate layout, the build pipeline, and how to add a new one. Read this before creating a new WASM crate or changing how an existing one builds.*

---

## When a WASM crate is the right answer

Tugdeck is a JS/TypeScript app. Most parsing, formatting, and rendering work stays in JS — adding a WASM crate has real costs (build complexity, async init, an extra crate to maintain, a few hundred KB of bundle weight). A WASM crate earns its place only when **a clear speed-or-correctness win justifies those costs**:

- **Markdown lexing / parsing.** `tugmark-wasm` wraps `pulldown-cmark`. Block-level lex is hot during streaming render; pulldown-cmark is the fastest correct CommonMark+GFM lexer available.
- **Diff computation.** `tugdiff-wasm` wraps `imara-diff`. 10–30× faster than JS alternatives on large inputs; pathological-input safe via histogram + Myers heuristics.

If a candidate workload doesn't have that profile (ANSI parsing, JSON-tree rendering, KaTeX, Mermaid), it stays in JS. See [tide-assistant-rendering.md `[D06]`](../roadmap/tide-assistant-rendering.md#d06-wasm-where-earns) for the full rationale.

## The crates

| Crate | Wraps | Exports | Lazy-loaded? |
|---|---|---|---|
| `tugmark-wasm` | `pulldown-cmark` | `lex_blocks`, `parse_to_html`, `parse_blocks_to_html` | No — Tide card boot dependency |
| `tugdiff-wasm` | `imara-diff` | `parse_unified_diff`, `two_text_diff` | Yes — first `DiffBlock` mount |

Both live at `tugdeck/crates/<name>/` and follow the same conventions below.

## Per-crate layout

```
tugdeck/crates/<name>/
├── .gitignore           # one line: "target/"
├── Cargo.toml
├── Cargo.lock           # checked in via `git add -f` (root .gitignore excludes Cargo.lock globally)
├── README.md            # API shape, build/test commands, "why WASM" pointer
├── src/
│   └── lib.rs
└── pkg/                 # built artifact, checked in via `git add -f`
    ├── .gitignore       # auto-generated by wasm-pack: "**" (the reason -f is needed)
    ├── package.json
    ├── <name>_bg.wasm
    ├── <name>_bg.wasm.d.ts
    ├── <name>.d.ts
    └── <name>.js
```

**`Cargo.toml` shape** (canonical):

```toml
[package]
name = "<name>"
version = "0.1.0"
edition = "2021"

[lib]
crate-type = ["cdylib", "rlib"]

[dependencies]
wasm-bindgen = "0.2"
# ... the wrapped Rust crate, plus serde + serde-wasm-bindgen if returning structured JS values

[profile.release]
opt-level = 3
lto = true
```

The `cdylib` target is what wasm-pack consumes; the `rlib` target lets `cargo test` link plain-Rust unit tests against the crate's non-`#[wasm_bindgen]` core. Both are needed — drop `rlib` and `cargo test` cannot link the lib; drop `cdylib` and wasm-pack cannot build the wasm module.

**Workspace registration: none.** WASM crates are *not* members of `tugrust/Cargo.toml`'s workspace. They build with their own toolchain target (`wasm32-unknown-unknown`) and pull a different set of dependencies (wasm-bindgen, serde-wasm-bindgen). Including them in the workspace would force every `cd tugrust && cargo build` to also build wasm targets, slowing native development for no benefit. Keep them standalone.

## Build pipeline

### Single recipe at the repo root

```sh
just wasm
```

This invokes `wasm-pack build --target web --release` for every crate under `tugdeck/crates/`. Adding a new crate means appending one line to the recipe in `Justfile`:

```just
wasm:
    #!/usr/bin/env bash
    set -euo pipefail
    WASM_PACK="$(command -v wasm-pack 2>/dev/null || echo "$HOME/.cargo/bin/wasm-pack")"
    ...
    "$WASM_PACK" build --target web --release tugdeck/crates/tugmark-wasm
    "$WASM_PACK" build --target web --release tugdeck/crates/tugdiff-wasm
    "$WASM_PACK" build --target web --release tugdeck/crates/<new-crate>   # ← one new line
```

`just app` depends on `wasm`, so anything that rebuilds the Mac app rebuilds the WASM artifacts first.

### Why pre-built artifacts are checked in

`pkg/` is committed so that:
- A clean `bun run dev` works without requiring contributors to install `wasm-pack` first.
- The Tug.app build phase doesn't need a Rust toolchain available at codesign time.
- HMR doesn't pause for a multi-second Rust rebuild on a file the developer didn't touch.

The cost: every `just wasm` produces fresh `pkg/` bytes (compiler-version-dependent), and `git status` will show them dirty. Re-add with `git add -f tugdeck/crates/<name>/pkg/` when you actually want the new bytes committed; otherwise `git restore` discards them.

### Vite watcher exclusions

Each crate's `pkg/` is excluded from Vite's file watcher in `tugdeck/vite.config.ts`:

```ts
watch: {
  ignored: ["**/palette-engine.ts", "**/tugmark-wasm/pkg/**", "**/tugdiff-wasm/pkg/**"],
},
```

Without this, every `just wasm` rebuild would trigger a full HMR reload mid-session.

## Tests

WASM crates test their **plain-Rust core**, not the `#[wasm_bindgen]` shells:

```sh
cd tugdeck/crates/<name>
cargo test
```

Pattern: write the work as a regular Rust function (e.g. `parse_unified_diff_text(&str) -> Vec<DiffHunk>`), test it with native unit tests, and write the `#[wasm_bindgen]` export as a thin shell that calls it and serializes the result. This keeps the tests fast (no wasm runtime) and the test surface honest (the algorithm, not the FFI).

WASM crates do *not* run from the tugrust workspace test runner; `cd tugrust && cargo nextest run -p <name>` will fail with "package ID specification did not match." That's expected — they have their own toolchain target.

The repo-wide `-D warnings` policy applies; check with `cargo clippy --all-targets -- -D warnings` from the crate directory.

## Runtime: lazy loading vs. boot dependency

Treat each crate's load timing as a deliberate choice:

- **Boot dependency** — the WASM module loads on tugdeck startup. Use only when the crate is on the critical-render path. `tugmark-wasm` qualifies (every assistant turn renders markdown).
- **Lazy** — the module loads on first feature encounter, with a placeholder during fetch. The default for new crates. `tugdiff-wasm` is lazy-loaded by `tugdeck/src/lib/lazy/load-tugdiff-wasm.ts` on the first `DiffBlock` mount.

Lazy is preferred because it bounds the boot bundle: a session that never sees a feature pays nothing for it. See [tide-assistant-rendering.md `[D10]`](../roadmap/tide-assistant-rendering.md#d10-lazy-load-stretch).

## Adding a new WASM crate (checklist)

1. Confirm WASM is justified (speed or correctness win; not "Rust is nice"). Cite the decision somewhere — a `[D##]` in the relevant roadmap plan or an updated entry in this doc.
2. Create `tugdeck/crates/<name>/` with `Cargo.toml`, `src/lib.rs`, `README.md`, `.gitignore` (one line: `target/`).
3. Use the canonical `Cargo.toml` shape above. Remember `crate-type = ["cdylib", "rlib"]`.
4. Write the work as plain Rust functions; add `#[wasm_bindgen]` thin wrappers that delegate. Add `#[cfg(test)]` unit tests against the plain Rust functions.
5. Run `cargo test` and `cargo clippy --all-targets -- -D warnings` from the crate dir.
6. Append the crate to the `wasm` recipe in `Justfile` (one line).
7. Append `**/<name>/pkg/**` to the watcher `ignored` list in `tugdeck/vite.config.ts`.
8. Run `just wasm` once to produce `pkg/`. `git add -f tugdeck/crates/<name>/pkg/ tugdeck/crates/<name>/Cargo.lock` to track them.
9. Wire the consumer side. Default to a lazy loader at `tugdeck/src/lib/lazy/load-<name>.ts` unless the module is genuinely on the boot path.
10. Update this file's "The crates" table.

## Files in scope

| File | Role |
|---|---|
| `tugdeck/crates/<name>/Cargo.toml` | Crate manifest. `cdylib + rlib`, `wasm-bindgen`, release profile. |
| `tugdeck/crates/<name>/src/lib.rs` | Plain Rust core + `#[wasm_bindgen]` export shells. |
| `tugdeck/crates/<name>/pkg/` | wasm-pack output. Checked in via `git add -f`. |
| `tugdeck/crates/<name>/README.md` | Per-crate API shape and build/test pointer. |
| `Justfile` `wasm` recipe | Single-recipe driver — one `wasm-pack build` line per crate. |
| `Justfile` `app` recipe | Depends on `wasm`; ensures Tug.app builds always include current WASM. |
| `tugdeck/vite.config.ts` | Watcher `ignored` list — one entry per `pkg/`. |
| `tugdeck/src/lib/lazy/load-<name>.ts` | Lazy-loader, when the crate is not a boot dependency. |
